# AIPA MCP Governance Overlay v0.1 Demo Walkthrough

This walkthrough gives reviewers a fast path through the v0.1 research milestone.

It is designed for someone who wants to understand the repository in roughly 10 minutes without reading every file.

## What this project demonstrates

This repository demonstrates a complementary AIPA governance overlay for MCP-style agent workflows.

The core idea is:

```text
Security controls what agents can do.
Governance records explain why actions were allowed, denied, or left unsupported.
```

The repository does not replace an MCP runtime or security platform. It adds reviewable governance artifacts around MCP-related decisions.

## Quick start

From the repository root, run:

```bash
python validator/aipa-governance/run_demo_validation.py
```

This command validates the demo artifact set and scenario folders listed in:

```text
examples/aipa-governance/validation-manifest.json
```

The manifest separates:

```text
structural validation
expected governance outcome
```

That means a scenario can be structurally valid even when its expected governance outcome is `FAIL` or `UNSUPPORTED`.

## Step 2: inspect the scenario index

Open:

```text
examples/aipa-governance/scenarios/README.md
```

This file lists all shipped scenario packages.

v0.1 includes six scenarios:

| Scenario | Expected outcome | Review focus |
| --- | --- | --- |
| Filesystem Write Review | PASS | Runtime tool-use governance for a scoped filesystem change request. |
| Browser Search Source Review | PASS | Source identifiers, citation requirements, and unsupported-claim boundaries. |
| Database Read-Only Query Review | PASS | Scoped read-only database access, approved query scope, and result handling. |
| MCP Server Install Review | PASS | Server lifecycle approval governance. |
| Denied MCP Server Install | FAIL | Governance-supported denial when required trust or scope evidence is insufficient. |
| Unsupported Verification Boundary | UNSUPPORTED | Evidence-bound review when required external proof is outside the available boundary. |

## Step 3: inspect one PASS scenario

Recommended starting point:


Review these files in order:

Expected governance outcome:

```text
PASS
```

## Step 4: inspect one FAIL scenario

Open:

```text
examples/aipa-governance/scenarios/mcp-server-install-denied/
```

This scenario is intentionally designed to produce:

```text
FAIL
```

The important point is that the files can still be structurally valid.

The governance outcome is `FAIL` because the review package supports denial of the request.

That is different from broken JSON or a broken scenario folder.

## Step 5: inspect the UNSUPPORTED scenario

Open:

```text
examples/aipa-governance/scenarios/unsupported-verification-boundary/
```

This scenario is designed to produce:

```text
UNSUPPORTED
```

It demonstrates the case where a reviewer cannot honestly return `PASS` or `FAIL` because required evidence is outside the available verification boundary.

This is important because governance systems should not force false certainty.

## Step 6: inspect the validator

Open:

```text
validator/aipa-governance/validate_governance_blocks.py
validator/aipa-governance/run_demo_validation.py
validator/aipa-governance/README.md
```

The validator is intentionally simple. It checks MVP structure and consistency, not live runtime behavior.

It validates artifacts such as:

- Governance Knowledge Blocks
- execution receipts
- server trust profiles
- install governance records
- policy blocks
- runtime scenario folders
- server install scenario folders
- unsupported verification boundary scenarios

## Step 7: understand the boundary

This repository does not claim:

- official ToolHive or Stacklok integration
- runtime enforcement
- production certification
- cryptographic proof
- external attestation
- live MCP registry integration
- live policy engine integration
- third-party endorsement

It is a research prototype showing how governance evidence can be structured around MCP-related decisions.

## What to look for as a reviewer

A reviewer should ask:

- Is the governance claim clear?
- Is the policy reference present?
- Is the policy fingerprint present?
- Is the evidence package understandable?
- Is the verification boundary explicit?
- Is the expected outcome reasonable?
- Are unsupported claims separated from reviewable claims?
