# AIPA MCP Governance Research Overlay v0.2 Roadmap

This roadmap describes the next research phase after the v0.1 milestone.

v0.1 proved that AIPA governance artifacts can be organized into reviewable scenario packages across `PASS`, `FAIL`, and `UNSUPPORTED` outcomes.

v0.2 should make the project easier to review, easier to demonstrate, and easier to evaluate as a complementary governance layer for MCP-style agent workflows.

## v0.2 objective

Move the repository from:

```text
credible research prototype
```

to:

```text
partner-reviewable governance reference package
```

The goal is not runtime integration yet. The goal is clarity, repeatability, reviewability, and partner-safe positioning.

## Priority 1: demo walkthrough

Add a reviewer-friendly walkthrough that explains how to inspect the project in 10 minutes.

Planned file:

```text
docs/aipa-governance/v0.1-demo-walkthrough.md
```

The walkthrough should explain:

- where to start
- how to run validation
- how to inspect the six scenarios
- how to understand `PASS`, `FAIL`, and `UNSUPPORTED`
- what the project does not claim

## Priority 2: architecture diagrams

Add simple diagrams showing how AIPA governance artifacts sit beside MCP-style runtime and security systems.

Planned file:

```text
docs/aipa-governance/architecture-diagrams.md
```

The diagrams should cover:

- runtime versus governance layer
- governance artifact flow
- server install lifecycle
- verification boundary handoff
- audit package generation

Use simple Mermaid diagrams where possible.

## Priority 3: partner review package

Add a document that explains the project in partner-safe language for maintainers, security engineers, and governance reviewers.

Planned file:

```text
docs/aipa-governance/partner-review-package.md
```

The document should emphasize:

- complementary governance layer
- non-competitive positioning
- no official integration claim
- governance evidence around MCP decisions
- potential value to agent-security ecosystems

## Priority 4: external verifier handoff model

Add a model showing how AIPA governance records could hand review packages to independent verification systems without merging trust surfaces.

Planned files:

```text
docs/aipa-governance/external-verifier-handoff.md
examples/aipa-governance/verifier-handoff-package.json
```

The model should separate:

```text
governance declaration
```

from:

```text
independent verification
```

This supports the AIPA boundary principle that governance records declare claims while verification systems test evidence.

## Priority 5: policy fingerprint hardening

Add notes describing how placeholder policy fingerprints can evolve into canonical policy identifiers.

Planned files:

```text
docs/aipa-governance/policy-fingerprints.md
examples/aipa-governance/policy-fingerprint-example.json
```

The document should explain:

- why policy fingerprints matter
- what is currently placeholder-only
- what canonicalization would require
- what should remain out of scope for v0.2

## Priority 6: reusable overlay template

Add a template plan for repeating the AIPA governance overlay pattern across other public AI infrastructure repositories.

Planned file:

```text
docs/aipa-governance/reusable-overlay-template.md
```

The template should define:

- minimum documentation set
- minimum schema set
- minimum examples
- minimum validator behavior
- partnership-safe language requirements
- fork selection criteria

## v0.2 non-goals

v0.2 should not attempt to provide:

- official ToolHive integration
- production runtime enforcement
- live MCP registry integration
- commercial certification
- cryptographic proof system
- external attestation service
- enterprise deployment tooling

Those may be future research directions, but they are not required for v0.2.

## v0.2 success criteria

v0.2 is successful if a reviewer can understand:

- what the project is
- what it is not
- how to run validation
- how the scenarios demonstrate governance review
- how AIPA artifacts complement MCP-style runtimes
- how external verification could receive evidence without trusting the originating governance layer
- how the overlay pattern could be repeated safely across compatible public repositories

## Status

This roadmap is a planning document. It does not claim that any v0.2 feature is complete until implemented in the repository.
