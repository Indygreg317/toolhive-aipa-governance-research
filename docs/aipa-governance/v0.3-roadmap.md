# AIPA MCP Governance Research Overlay v0.3 Roadmap

This roadmap describes the next research phase after the v0.2 partner-reviewable milestone.

v0.2 established the AIPA MCP governance overlay as a clear, repeatable, partner-safe reference package.

v0.3 should move the project from:

```text
partner-reviewable governance reference package
```

to:

```text
more testable and reusable governance overlay toolkit
```

The goal is still not runtime integration.

The goal is stronger examples, clearer validation, better template reuse, and more precise evidence handling.

## v0.3 objective

Improve the repository so reviewers can evaluate not only the documents, but also the repeatability of the overlay pattern.

v0.3 should make it easier to answer:

- Can a new scenario be created from the templates?
- Can the validator detect common structural mistakes?
- Can policy fingerprint consistency be checked more clearly?
- Can review handoff packages be assembled predictably?
- Can unsupported claims be separated from reviewable claims every time?

## Priority 1: template instantiation example

Add one complete example created directly from the reusable templates.

Planned output:

```text
examples/aipa-governance/scenarios/template-instantiated-example/
```

The scenario should show how a reviewer can go from template to concrete package.

It should include:

- request or install request
- decision context or trust profile
- receipt or lifecycle decision
- governance record
- verification boundary map
- audit package summary
- optional review handoff package

## Priority 2: template usage guide

Add a short guide explaining how to create a new governance scenario from the templates.

Planned file:

```text
docs/aipa-governance/template-usage-guide.md
```

The guide should include:

- choosing the right template
- replacing placeholders
- keeping policy fingerprints consistent
- declaring unsupported claims
- adding the scenario to the manifest when appropriate
- running validation

## Priority 3: validator hardening

Improve validator checks without turning the project into a runtime enforcement system.

Possible checks:

- required artifact paths exist
- template placeholder values are not used in finalized examples
- expected governance outcome is valid
- policy fingerprint is present
- policy fingerprint is consistent across scenario artifacts
- unsupported claims are present when expected outcome is `UNSUPPORTED`
- high-risk scenarios require human oversight

## Priority 4: policy fingerprint example

Add a concrete policy fingerprint example that shows the difference between:

```text
placeholder fingerprint
```

and:

```text
canonical policy fingerprint candidate
```

Planned file:

```text
examples/aipa-governance/policy-fingerprint-example.json
```

This should remain a research example, not a cryptographic proof claim.

## Priority 5: review handoff index

Add an index that explains what an external reviewer receives and how to inspect it.

Planned file:

```text
docs/aipa-governance/review-handoff-index.md
```

The index should map:

- governance record
- evidence references
- policy reference
- policy fingerprint
- verification boundary map
- audit package summary
- unsupported claims
- expected outcome

## Priority 6: overlay reuse criteria

Add criteria for choosing future public repositories where this overlay pattern may be useful.

Planned file:

```text
docs/aipa-governance/overlay-reuse-criteria.md
```

Criteria may include:

- public governance relevance
- agent/tool-use surface
- clear policy or review boundary
- non-competitive partner-safe positioning
- ability to demonstrate value without modifying runtime behavior
- ability to keep claims evidence-bound

## v0.3 non-goals

v0.3 should not attempt to provide:

- official ToolHive integration
- production runtime enforcement
- live MCP registry integration
- commercial certification
- cryptographic proof system
- external attestation service
- enterprise deployment tooling
- claims of Stacklok endorsement or partnership

## v0.3 success criteria

v0.3 is successful if a reviewer can:

- create a new scenario from a reusable template
- understand how the validator checks structural correctness
- see policy fingerprint consistency in a concrete example
- inspect a review handoff package with less ambiguity
- distinguish reviewable claims from unsupported claims
- understand when this overlay pattern is appropriate for another repository

## Status

This roadmap is planning only.

It does not claim that any v0.3 feature is complete until implemented in the repository.
